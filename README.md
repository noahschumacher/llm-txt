# llms.txt Generator

A web application that automatically generates `llms.txt` (and optionally `llms-full.txt`) files for any given website by crawling its structure, extracting content, and optionally using an LLM to produce high-quality, human-readable descriptions. Specific prompt can be seen in ProjectPrompt.md

## What is llms.txt?

`llms.txt` is a proposed standard (see [llmstxt.org](https://llmstxt.org)) that helps Large Language Models better understand and navigate website content. Similar to `robots.txt` for search engine crawlers, `llms.txt` provides a curated, structured Markdown file that signals to AI systems which pages matter most and what they contain.

The format is intentionally simple:
- An `H1` with the site name
- A blockquote with a one-sentence site description
- `H2` sections grouping related pages
- Each entry as `[Page Title](url): Brief description`

There is also a `llms-full.txt` variant that includes the full content of each page concatenated into a single document, useful for IDE integrations and agent workflows that can handle large inputs.

## How It Works

### User Flow

The user visits the web app, enters a URL, selects a generation mode, and receives a generated `llms.txt` file they can copy or download. Generation progress is streamed in real time via Server-Sent Events (SSE) so the user sees pages being processed as they complete.

### Generation Modes

**Basic** ‚Äî Crawls the site and builds `llms.txt` using page titles and `<meta name="description">` content as descriptions. Fast, no external API calls required.

**Enhanced** ‚Äî Same crawl, but each page's description is generated by an LLM (using the scraped body content as input). Produces significantly better output ‚Äî concise, meaningful summaries rather than SEO copy. Requires a configured LLM API key on the server.

**Full Text** ‚Äî An optional add-on to either mode. In addition to `llms.txt`, generates `llms-full.txt` by concatenating all crawled page content as Markdown under their respective section headers.

### Request Flow

1. User submits a URL with selected mode options via the frontend form.
2. The Go backend receives the request and begins the generation pipeline.
3. **URL validation** ‚Äî normalize the input (add scheme if missing, lowercase host, strip trailing slashes), reject malformed URLs early.
4. **`robots.txt` fetch** ‚Äî download and parse the site's `robots.txt` to build a disallow list; all subsequent crawl requests respect this.
5. **Sitemap discovery** ‚Äî attempt to fetch `sitemap.xml` (and follow `sitemap_index.xml` if present). Parse sitemap URLs along with any `<priority>` and `<lastmod>` hints to seed the crawl queue intelligently. Fall back to BFS starting from the root URL if no sitemap exists.
6. **BFS crawl loop** ‚Äî process the queue with a configurable depth limit (default: 3 levels) and max page cap (default: 50 pages):
   - Skip URLs disallowed by `robots.txt` or already visited
   - `GET` each page, follow redirects, skip non-HTML content types
   - Parse HTML: extract `<title>`, `<meta name="description">`, `<h1>`, and main body text
   - Extract internal `<a href>` links and add unseen ones to the queue
   - Enforce a polite crawl delay between requests
7. **Deduplication and filtering** ‚Äî normalize URLs (strip fragments, canonicalize query params), drop duplicates, filter out low-value pages (e.g. tag pages, pagination).
8. **Section inference** ‚Äî group crawled pages into logical sections based on URL path segments (e.g. `/docs/*` ‚Üí Documentation, `/api/*` ‚Üí API Reference, `/blog/*` ‚Üí Blog). These become the `H2` sections in the output.
9. **Description generation**:
   - *Basic mode*: use `<meta name="description">` as the description field, falling back to a truncated `<h1>` + first paragraph.
   - *Enhanced mode*: for each page, send the title and extracted body text to the LLM with a prompt instructing it to write a single concise sentence describing the page's purpose. Calls are dispatched concurrently using a bounded goroutine pool (semaphore pattern via `golang.org/x/sync/errgroup`) to control parallelism and respect LLM rate limits. Results are streamed back to the client via SSE as each page completes.
10. **Site-level description** *(enhanced mode)* ‚Äî after page descriptions are generated, send a summary of the top-level pages to the LLM to produce the one-sentence site description used in the blockquote.
11. **`llms.txt` assembly** ‚Äî format the output per spec: `H1` site name, blockquote description, `H2` sections, and `[Title](url): Description` entries per page.
12. **`llms-full.txt` assembly** *(if requested)* ‚Äî concatenate all crawled page body content as Markdown, organized under the same section headers.
13. **Response** ‚Äî return both files as text to the frontend. The UI displays each in a copy/download-ready text area, along with a summary of pages crawled, sections detected, and mode used.

## Architecture

### Backend (Go)

The application is a single Go binary that serves both the static frontend and handles all API requests. There are no separate frontend build steps or servers.

**Key packages and responsibilities:**

- `main.go` ‚Äî loads env vars via `pkg`, initializes dependencies, wires the server, handles graceful shutdown
- `server/` ‚Äî HTTP server struct, route registration, and handler methods (one file per domain); middleware and services live in subpackages
  - `server/middleware/` ‚Äî request-scoped middleware (logging, timeout, recovery)
  - `server/services/generator/` ‚Äî orchestrates the crawl ‚Üí describe ‚Üí format pipeline; called by the handler
- `crawler/` ‚Äî robots.txt parsing, sitemap fetching, BFS crawl loop, HTML parsing and content extraction
- `clients/llm/` ‚Äî LLM API client abstraction, prompt construction, concurrent dispatch with semaphore pool, response parsing
- `pkg/` ‚Äî env var loading helpers (`LoadStringEnv`, `LoadIntEnv`, `LoadBoolEnv`, etc.) and shared utilities

The frontend HTML/CSS/JS is embedded directly into the binary using Go's `//go:embed` directive, keeping deployment a single binary with no external asset dependencies.

### Frontend

A single static HTML page with vanilla JS. No framework, no build step. The form submits to the backend generation endpoint and opens an SSE connection to stream progress updates. Once complete, the generated file content is rendered into copy/download-ready text areas.

### LLM Integration

The LLM client is configured server-side via environment variable. The API key is never exposed to the frontend. The provider and model are configurable ‚Äî a cheap, fast model (e.g. `claude-haiku-4-5` or `gpt-4o-mini`) is appropriate since the prompts are short and the task is straightforward. Per-page calls are batched via a concurrent worker pool rather than sequentially or in one giant prompt, giving a good balance of quality, speed, and cost.

## Configuration

The application is configured via environment variables:

| Variable | Description | Required |
|---|---|---|
| `PORT` | Port to listen on (default: `8080`) | No |
| `LLM_PROVIDER` | LLM provider (`anthropic` or `openai`) | Enhanced mode only |
| `LLM_API_KEY` | API key for the LLM provider | Enhanced mode only |
| `LLM_MODEL` | Model to use (e.g. `claude-haiku-4-5`, `gpt-4o-mini`) | No (has default) |
| `CRAWL_MAX_PAGES` | Maximum pages to crawl per request (default: `50`) | No |
| `CRAWL_MAX_DEPTH` | Maximum BFS depth (default: `3`) | No |
| `CRAWL_DELAY_MS` | Delay between requests in ms (default: `500`) | No |
| `LLM_CONCURRENCY` | Max concurrent LLM calls (default: `5`) | No |

## Setup and Development

### Prerequisites

- Go 1.21+

### Running Locally

```bash
# Clone the repo
git clone https://github.com/your-username/llms-txt-generator
cd llms-txt-generator

# Copy and fill in environment variables
cp .env.example .env

# Run
go run ./...
```

The app will be available at `http://localhost:8080`.

### Building

```bash
go build -o llms-txt-generator ./...
```

The resulting binary is fully self-contained ‚Äî the frontend is embedded and no additional files are needed.

## Deployment (Sevalla)

The app deploys as a single Go binary on Sevalla.

1. Connect your GitHub repo in the Sevalla dashboard
2. Set the build command to `go build -o server ./...`
3. Set the start command to `./server`
4. Add the environment variables from the table above under Settings ‚Üí Environment Variables
5. Deploy

Sevalla will handle HTTPS termination. No additional configuration is needed.

## Notes on AI Usage

This project uses LLM APIs (Anthropic or OpenAI) in enhanced mode to generate page descriptions and the site-level summary. The API key is stored as a server-side environment variable and is never sent to the client. All LLM prompts are constructed server-side.

The generated `llms.txt` output is meant to be reviewed and potentially edited by the site owner before publishing ‚Äî think of this tool as a high-quality first draft, not a fully automated pipeline.
